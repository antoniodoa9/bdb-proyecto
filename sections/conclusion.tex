\subsection{Conclusions}
% ===========================================================================

This project started from a question that looked simple: can a medical database be GDPR-compliant by design and, at the same time, fast enough for a real clinical setting? After developing the full system, our conclusion is yes ---but with one important condition: the key decisions must be made in the data schema itself, and not delegated to the application code or to a permission layer that could be bypassed. The project also left us with a second lesson we were not looking for at the start: there is no single \enquote{good} database, only the one that best fits each kind of data. That is why we ended up combining three paradigms: the relational one, the XML document one and the NoSQL document one.

The first idea we confirmed is that privacy-by-design is something tangible and not just a label. Splitting \texttt{paciente\_anonimo} from \texttt{datos\_personales} with a SHA-256 hash works because the anonymisation cannot be undone: the \texttt{investigador} user is denied access to the personal data in the engine itself, not in the application. Recovering the identity from a \texttt{codigo\_hash} would require either the original ID (which is not stored in the database) or breaking SHA-256 by brute force, which is infeasible today. The \texttt{ON DELETE RESTRICT} constraints reinforce that guarantee by preventing a clinical record from being deleted while its personal file still exists.

As for performance, the decision to assign a different engine to each table paid off and, above all, we were able to quantify it. Indexes made very noticeable differences: a patient's history join went from 39.89~ms to 0.64~ms (62.3 times faster) and the monthly event count from 46.06~ms to 1.36~ms (33.8×). The ARCHIVE engine compressed the sick-leave archive from 2,560~KiB to 533~KiB, 79.2\% less, by dropping mechanisms that an immutable historical archive does not need. And MEMORY kept the triage queue at 0.77~ms, 45.9\% below InnoDB, with the added benefit of a constant latency that is independent of the disk. The fact that these figures come from synthetic data does not make them any less valid: the generator produces clinically coherent histories, so from a query point of view they behave like real data.

The second part showed that, when data is hierarchical, storing it as an XML tree is far more natural than splitting it into tables. The three XQuery queries ended up working as a small clinical decision-support system: one crosses a patient's emergency visit, history and sick leave in a single query; another detects chronic patients ---and here we found that relying on the document's structure turns a query that was driving memory usage up into one of linear cost---; and the last one generates hypoxia alerts, where the most delicate part was handling type conversions, since in XML everything is text. The lesson is clear: the effort invested in giving the documents a good shape pays off handsomely when querying them.

The third part confirmed that MySQL and MongoDB can coexist without interfering, each taking care of what it does best: the transactional side in the relational engine and the flexible clinical knowledge ---the guidelines--- in the document store. We draw three conclusions from this phase. First, that privacy-by-design is feasible in practice: since the ID becomes a hash at insertion time, the 50,000 records reached MongoDB already anonymised, with no exposure window. Second, that MongoDB's \texttt{.explain()} reproduced the same indexing lesson we had already observed in MySQL: the index on \texttt{anio\_nacimiento} turned a scan of all 50,000 documents into an index scan that only examines the 12,040 relevant ones. And third, perhaps the most relevant from an engineering standpoint, that delegating the heavy processing to the engine (the aggregation pipelines) and keeping Python for the presentation layer is what distinguishes a solution that scales from one that stops working as soon as the data grows.

All in all, the main lesson we take away is that the tension between GDPR compliance and efficiency was a false dilemma: when they are well designed, both goals reinforce each other instead of getting in the way. And that choosing a storage technology is not a matter of trends, but of matching the tool to the shape of the data: tables for the tabular, XML trees for the hierarchical and documents for the flexible. Handling three technologies at once does add complexity, but each one contributes exactly where the others fall short; and deploying all of them in Docker containers made it considerably easier to reproduce the same environment across the three members of the group.

\subsection{Limitations}
% ===========================================================================

It is worth being honest about the scope of what we achieved. The most obvious limitation is that the whole system was validated with synthetic data: although the generator produces clinically coherent histories, they are still artificial, so the work does not include validation with real patient information or with clinical professionals assessing the usefulness of the queries in practice. Along the same lines, the catalogue of clinical guidelines that feeds the MongoDB part is small ---twelve protocols---, enough to prove the concept but far from the volume a production system would handle.

The technical scope should also be qualified. All the tests were run on a single machine and a single instance of each engine, so the performance figures are representative of that setting, but not of a distributed deployment or of the real concurrency of many simultaneous users. The graphical interface, in turn, is a functional prototype meant to illustrate the integration of the three technologies, not an application ready to be used in a hospital. In short, the project should be understood as a solid, measured proof of concept rather than a finished product.

\subsection{Future lines of research}
% ===========================================================================

Even though the system does what we set out to do, several paths came up during development that we would have liked to follow and that we leave noted here for the future.

The most natural one would be moving on to graph databases (such as Neo4j). Many clinical questions are, deep down, questions about networks: which diseases appear together, which drugs interact with each other or how a contagion spreads. All of that is expressed comfortably by traversing a graph, whereas in SQL it forces you to chain costly joins. Projecting the \texttt{evento\_clinico} table onto a graph and evaluating algorithms such as shortest path would be a logical continuation of this work.

Another pending line is free clinical text. We now structure events well, but handwritten notes, imaging reports and discharge summaries are still the greatest challenge. It would be interesting to exploit MongoDB's full-text search ---or to lean on language models--- to perform semantic searches over that content, always keeping the hash-based anonymisation we already have in place.

There is also everything related to scalability. At the moment we work with a single MySQL server; a real hospital deployment would need replicas, and there an interesting open question remains: what happens to the MEMORY tables, which reside in each node's RAM, once several nodes are involved.

On the security side, encrypting the ARCHIVE tables was left pending. InnoDB has tablespace encryption well sorted out, but in ARCHIVE it is not so direct, so it would be worth studying how to protect that data at rest without sacrificing the compression that worked so well for us.

Finally, with 40,000 patients there is plenty of material to train simple predictive models on the anonymised data: readmission risk, estimated length of stay or early warnings from the triage vital signs. It would be a way to close the loop and show a concrete application of the whole GDPR-compliant architecture we designed.
