\subsection{Conclusions}
% ===========================================================================

This project started from a question that sounded simple: can a medical database be GDPR-compliant by design and, at the same time, fast enough to be used in a real clinical setting? After building everything we built, our answer is yes ---but with one condition: the important decisions have to be made in the data schema itself, and not left to the application code or to a permission layer that anyone could bypass. And, almost without looking for it, the project ended up teaching us something else: there is no single \enquote{good} database, only the one that best fits each kind of data. That is why we ended up combining three of them: the relational one, the XML document one and the NoSQL document one.

The first thing we confirmed is that privacy-by-design is not just a slogan. Splitting \texttt{paciente\_anonimo} from \texttt{datos\_personales} with a SHA-256 hash really works because the anonymisation cannot be removed: we closed the \texttt{investigador} user's access to the personal data inside the engine itself, not in the application. Going back from a \texttt{codigo\_hash} to a real person would require either the original ID (which is not stored in the database) or breaking SHA-256 by brute force, something that is infeasible today. On top of that, the \texttt{ON DELETE RESTRICT} constraints make sure no clinical record can be deleted while its personal file still exists.

On performance, betting on a different engine for each table paid off and, above all, we were able to measure it. Indexes made a huge difference: a patient's history join dropped from 39.89~ms to 0.64~ms (62.3 times faster) and the monthly event count from 46.06~ms to 1.36~ms (33.8×). ARCHIVE compressed the sick-leave archive from 2,560~KiB to 533~KiB, 79.2\% less, simply by giving up things a historical archive does not need. And MEMORY kept the triage queue at 0.77~ms, 45.9\% below InnoDB, with the added bonus of a constant latency that does not depend on the disk. The fact that all these numbers come from synthetic data does not make them any less valid: the generator produces clinically coherent histories, so from a query point of view they behave just like real data.

The second part taught us that, when data is hierarchical, storing it as an XML tree is far more natural than chopping it into tables. The three XQuery queries ended up working as a small decision-support system: one crosses a patient's emergency visit, history and sick leave in a single query; another detects chronic patients ---and here we learned that leaning on the document's own structure turns a query that was blowing up our memory into one that runs comfortably---; and the last raises hypoxia alerts, where the trickiest part was being careful with type conversions, because in XML everything is text. The lesson is clear: the time you invest in giving the documents a good shape pays off handsomely when you query them.

The third part confirmed that MySQL and MongoDB can coexist without stepping on each other, each doing what it does best: the transactional side in the relational engine and the flexible clinical knowledge ---the guidelines--- in the document store. We take away three ideas from it. First, that privacy-by-design is real: since the ID becomes a hash at insertion time, the 50,000 records reached MongoDB already anonymised, with no exposure window at all. Second, that MongoDB's \texttt{.explain()} repeated the very same indexing lesson we had already seen in MySQL: the index on \texttt{anio\_nacimiento} turned a scan of all 50,000 documents into an index scan that only looks at the 12,040 that matter. And third, perhaps the most important one from an engineering standpoint, that leaving the heavy lifting to the engine (the aggregation pipelines) and using Python only to display the result is what separates a solution that scales from one that collapses as soon as the data grows.

All in all, what we mostly take from the project is that the tension between GDPR compliance and efficiency was a false dilemma: when they are well designed, the two reinforce each other instead of getting in the way. And that choosing a storage technology is not about following trends, but about matching the tool to the shape of the data: tables for the tabular, XML trees for the hierarchical and documents for the flexible. Juggling three technologies does add complexity, true, but each one contributes exactly where the others fall short, and deploying all of them in Docker containers saved us a good deal of trouble when it came to setting up the same environment across the three of us.

\subsection{Future lines of research}
% ===========================================================================

Even though the system does what we set out to do, several paths came up during development that we would have liked to follow and that we leave noted here for the future.

The most natural one would be moving on to graph databases (such as Neo4j). Many clinical questions are, deep down, questions about networks: which diseases appear together, which drugs interact with each other or how a contagion spreads. All of that is expressed very comfortably by traversing a graph, whereas in SQL it forces you to chain costly joins. Projecting the \texttt{evento\_clinico} table onto a graph and trying out algorithms such as shortest path would be a logical continuation of this work.

Another pending line is free clinical text. We now structure events well, but handwritten notes, imaging reports and discharge summaries are still the hard nut to crack. It would be interesting to exploit MongoDB's full-text search ---or even to lean on language models--- to search that content semantically, always keeping the hash-based anonymisation we already have in place.

There is also everything related to scalability. Right now we work with a single MySQL server; a real hospital deployment would need replicas, and there an interesting open question remains: what happens to the MEMORY tables, which live in each node's RAM, once there are several nodes.

On the security side, we did not get to encrypting the ARCHIVE tables. InnoDB has tablespace encryption well sorted out, but in ARCHIVE it is not so direct, so it would be worth studying how to protect that data at rest without losing the compression that worked so well for us.

Finally, with 40,000 patients there is already plenty of material to train simple predictive models on the anonymised data: readmission risk, estimated length of stay or early warnings from the triage vital signs. It would be a way to close the loop and show a concrete application of the whole GDPR-compliant architecture we designed.
